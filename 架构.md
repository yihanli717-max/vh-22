# BizTrack 代码项目说明

## 1. 项目概览
BizTrack 是一个纯前端的多页面业务管理系统，主要用于帮助小商家管理：

- 产品信息（Products）
- 订单信息（Orders）
- 支出信息（Expenses / Finances）
- 业务概览仪表盘（Dashboard）

项目没有独立后端，也没有真实数据库。所有业务数据都保存在浏览器的 `localStorage` 中，因此部署方式比较轻，适合做静态站点部署。

从课程作业角度看，这个项目的优点是：

- 结构清晰，容易快速读懂
- 页面和功能足够多，容易找到 4 个具体缺陷
- 前端项目，容易补做 `Lighthouse`、`i18n`、`Cookie Banner`、`Privacy Policy`
- 后续上 `Vercel` 或 `Render` 的成本低

```text

├── README.md

├── orders.html         # 订单管理页

├── finances.html       # 支出管理页
├── finances.js         # 支出数据增删改查、搜索、排序、导出

```

## 3. 页面结构说明

### 3.1 Dashboard
- 文件：`index.html` + `script.js`
- 作用：
  - 展示总收入、总支出、余额、订单数量
  - 展示销售类别图表和支出图表
- 数据来源：
  - `bizTrackOrders`
  - `bizTrackTransactions`
  - `bizTrackProducts`
- 图表库：
  - 通过 CDN 引入 `ApexCharts`

### 3.2 Products 页面
- 文件：`products.html` + `products.js`
- 作用：
  - 添加、编辑、删除产品
  - 按列排序
  - 搜索产品
  - 导出产品 CSV
- 主要字段：
  - `prodID`
  - `prodName`
  - `prodDesc`
  - `prodCat`
  - `prodPrice`
  - `prodSold`

### 3.3 Orders 页面
- 文件：`orders.html` + `orders.js`
- 作用：
  - 添加、编辑、删除订单
  - 统计总收入
  - 按列排序
  - 搜索订单
  - 导出订单 CSV
- 主要字段：
  - `orderID`
  - `orderDate`
  - `itemName`
  - `itemPrice`
  - `qtyBought`
  - `shipping`
  - `taxes`
  - `orderTotal`
  - `orderStatus`

### 3.4 Finances 页面
- 文件：`finances.html` + `finances.js`
- 作用：
  - 添加、编辑、删除支出
  - 统计总支出
  - 按列排序
  - 搜索支出
  - 导出 CSV
- 主要字段：
  - `trID`
  - `trDate`
  - `trCategory`
  - `trAmount`
  - `trNotes`

### 3.5 Help / About 页面
- 文件：`help.html`、`help.js`、`about.html`
- 作用：
  - 展示项目使用说明
  - 展示开发者信息
- 特点：
  - 逻辑较少，主要是静态内容页

## 4. 数据流与存储方式
这个项目的“数据层”完全基于浏览器本地存储。

使用到的 `localStorage` key 包括：

- `bizTrackProducts`
- `bizTrackOrders`
- `bizTrackTransactions`

每个模块第一次加载时，会先检查对应 key 是否存在：

- 如果存在：读取本地数据
- 如果不存在：写入一组默认示例数据

这意味着：

- 数据只存在于当前浏览器
- 清空浏览器缓存后数据会丢失
- 不支持多用户协作
- 不支持服务端校验
- 不支持真正的权限控制

## 5. 代码组织特点

### 5.1 优点
- 采用“一个页面 + 一个脚本”的结构，初学者容易理解
- 产品、订单、支出的 CRUD 逻辑比较对称，便于统一重构
- 功能点集中，适合逐个改造并写成课程报告

### 5.2 当前存在的明显架构限制
- 没有模块化，逻辑函数都放在全局作用域
- 页面之间通过 `localStorage` 间接共享数据，没有统一的数据模型层
- 表单、渲染、排序、搜索、导出逻辑耦合在同一个文件中
- 很多 UI 行为依赖 `onclick` 和 `innerHTML`
- 没有测试体系

注意：
`没有测试` 更适合放进 baseline standards（80% coverage）部分，而不是作为 4 个 deficiency 之一。
